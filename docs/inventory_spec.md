# ТЗ: дашборд «Управление товарооборотом»

_Вкладка единой веб-платформы Маркона. Версия от 2026-05-13._

## 1. Цель и угол

Управление **паллетоместами** в трёх температурных камерах склада. Базовая логика: место в камере = ресурс, эффективная SKU = быстро освобождает паллеты, неэффективная = долго блокирует. Стоимостные метрики — вторичны.

## 2. Пользователи

- Руководитель склада (основной)
- Руководители Маркона
- Аналитики

Аудитория операционная: детализация важнее, чем «красивая агрегация».

## 3. Контекст

- 1 склад, **~300 SKU**.
- Ассортимент — ингредиенты для хот-догов, поставка на АЗС.
- **Сезонность ×3** в течение года → скользящие средние 7/14 дней как baseline не годятся, нужен год-к-году.
- Несколько температурных камер (стартовая конфигурация: две −18°C, одна +5°C, но **количество и ёмкости редактируются в настройках**). SKU привязан к конкретной камере. Метрики занятости считаются **отдельно по каждой камере**.

## 4. Источники данных

Выгрузки из 1С в **Excel**, загружаются через **UI самого дашборда** (по мере необходимости обновления данных, без cron / без прямого коннекта к 1С).

**Старт с дня 0:** в систему вбиваются начальные остатки на стартовую дату, дальше — приходы и расходы по дням.

Гранулярность модели данных: **день × склад × SKU**.

Три типа загружаемых отчётов из 1С:
1. **Стартовые остатки** (разово, на дату 0) — SKU + остаток в коробках.
2. **Поступления** (приходы) — SKU + дата + количество коробок.
3. **Отгрузки** (расходы) — SKU + дата + количество коробок.

Остатки на любую дату восстанавливаются из стартовых ± движения. Не нужно загружать ежедневный снимок остатков.

Поля справочника SKU (тоже из 1С, отдельной выгрузкой):
- Артикул, наименование
- Бренд / категория / поставщик
- Камера хранения
- Коробок на паллете
- Себестоимость / цена

**UI загрузки** на дашборде:
- Кнопки/зоны drop-area под каждый тип отчёта (стартовые остатки, поступления, отгрузки, справочник SKU).
- История загрузок (что когда было залито, кем).
- Идемпотентность: повторная загрузка того же файла не дублирует данные. Если в новой выгрузке есть запись на дату, уже бывшую в системе, — перезаписывается (либо валидируется и предупреждает).

## 5. Справочники и настройки дашборда

| Параметр | Тип | Дефолт |
|---|---|---|
| Камеры хранения | список с редактированием (название, температура, ёмкость в паллетоместах); можно добавлять/удалять | стартово 3: «−18 №1», «−18 №2», «+5» |
| Порог хвоста паллеты, % | редактируемое поле | **10%** |
| Окно для индекса «мёртвости», дней | редактируемое | **90** |
| Сезонный baseline | год-к-году / медианный профиль | год-к-году |

## 6. Метрики

### 6.1 Занятые паллетоместа SKU на дату

```
если остаток == 0:
    паллетомест = 0
иначе:
    полные = floor(остаток / коробок_на_паллете)
    хвост_% = (остаток % коробок_на_паллете) / коробок_на_паллете
    если полные == 0:
        паллетомест = 1
    иначе если хвост_% > порог:
        паллетомест = полные + 1
    иначе:
        паллетомест = полные
```

### 6.2 Утилизация камеры

```
утилизация_камеры_% = сумма(паллетомест_SKU в камере) / ёмкость_камеры
```

### 6.3 Паллето-дни SKU за период

```
паллето_дни = сумма(паллетомест_SKU за каждый день периода)
```

### 6.4 Индекс «мёртвости» SKU

```
индекс_мёртвости = паллето_дни за окно / коробок_отгружено за окно
```

Чем выше — тем «мертвее». Непрерывная шкала, без бинарных порогов. Сортировка → рейтинг.

_Граничный случай:_ если отгрузок 0 — индекс = ∞ (выносится в отдельную группу «совсем не отгружалось N дней»).

### 6.5 Оборачиваемость и дни покрытия

```
оборачиваемость = отгрузка_за_период / средний_остаток
дни_покрытия = текущий_остаток / среднесуточная_отгрузка
```

Среднесуточная отгрузка — **с поправкой на сезон** (по тому же окну прошлого года), не по последним 7/14 дням.

### 6.6 Lead time и точка заказа

Lead time **плавающий по SKU** — выводим из истории поступлений: среднее + стандартное отклонение. Точка заказа считается на каждый SKU отдельно.

### 6.7 ABC / XYZ

Строится с нуля по истории отгрузок (ABC — по объёму отгрузки, XYZ — по коэффициенту вариации).

## 7. Блоки дашборда

### Блок 1. Утилизация камер _(верхний экран)_

- Полосы загрузки трёх камер: текущая / средняя / пик за период.
- Дни до переполнения камеры при текущей динамике (поступления − отгрузки).
- Доля паллетомест, занятых SKU с высоким индексом мёртвости (топ-квартиль).

Первое, что видит пользователь: **где сейчас тесно и где скоро будет затор**.

### Блок 2. Динамика SKU

- Линии «остаток» и «отгрузка» по выбранной SKU.
- **Оверлей: тот же день прошлого года** (сезонный baseline).
- Подсветка вне коридора safety stock (когда выведем).

### Блок 3. Матрица эффективности SKU

- Scatter:
  - X — средняя дневная отгрузка
  - Y — средние занятые паллетоместа
  - Цвет — камера
  - Размер точки — паллето-дни накопительно за период
- **Обязательный фильтр по камере** (иначе три зоны в одном scatter — каша).

### Блок 4. Рейтинги SKU

- Топ по индексу мёртвости.
- Топ «остаток растёт, отгрузка падает» с поправкой на сезон.
- Топ по риску дефицита перед сезонным пиком (текущий остаток < прогнозируемая отгрузка следующего периода).

### Блок 5. Сезонный профиль и точка заказа SKU

- Отгрузка год-к-году, прогноз на ближайший месяц от прошлогоднего профиля.
- Расчётный safety stock и точка заказа с учётом lead time SKU.
- Сигнал «пора заказывать / запас в избытке к сезону».

## 8. Фильтры (глобальные на дашборде)

- Период (диапазон дат)
- Камера
- Категория / бренд / поставщик
- ABC-класс / XYZ-класс
- Конкретные SKU

## 9. Что НЕ делаем

- Метрика «недозагруженные паллеты / хвосты» — не нужна, склад их быстро убирает.
- Учёт срока годности — не нужен.
- Бинарные пороги «мёртвый / живой» — только непрерывная шкала.

## 10. Открытые вопросы

Закрыты на момент 2026-05-13. Точные ёмкости камер вносятся через UI настроек самим Киром при старте системы.
